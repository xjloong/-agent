# 多模态客服智能体技术文档

---

## 一、核心价值总览

### 1.1 赛题核心痛点

在大赛设定的多模态客服场景中，我们识别出以下 **5 个核心技术挑战**：

| 编号 | 痛点 | 具体表现 |
|------|------|----------|
| P1 | **多模态信息理解不准** | 用户上传图片（订单截图、故障照片、错误代码）时，纯文本模型无法准确提取关键信息，导致意图识别偏差 |
| P2 | **RAG 检索精度不足** | 40 本产品手册、20 万字知识库，简单关键词匹配无法定位到准确的段落，检索结果噪音大 |
| P3 | **多轮对话上下文丢失** | 用户追问"那别的型号呢？"时，传统截断策略丢失关键历史，智能体无法理解指代关系 |
| P4 | **大模型幻觉回答** | 模型在缺乏依据时编造操作步骤或产品参数，产生误导用户的风险 |
| P5 | **双语知识库融合困难** | 20 本中文 + 20 本英文手册，语言混杂时检索和回答的语言一致性难以保证 |

### 1.2 整体解决方案

我们设计了一套 **"多模态感知 → 意图路由 → 精准检索 → 记忆增强生成"** 四阶段管线，将视觉理解、向量检索、对话记忆和幻觉抑制深度整合为一个端到端的智能体系统。

```
用户输入(文本+图片)
       │
       ▼
┌─────────────────────────────────────────────────────┐
│  阶段一：多模态感知                                    │
│  Qwen-VL-Max 视觉编码 → 图文意图融合                   │
├─────────────────────────────────────────────────────┤
│  阶段二：意图路由                                      │
│  LLM 路由判定 → 直接回答 / 触发 RAG 检索                │
├─────────────────────────────────────────────────────┤
│  阶段三：精准检索                                      │
│  Embedding 向量化 → Milvus HNSW 检索 → ReRank 重排序   │
├─────────────────────────────────────────────────────┤
│  阶段四：记忆增强生成                                   │
│  摘要记忆注入 → DeepSeek 生成 → 幻觉抑制 → 图片自动关联  │
└─────────────────────────────────────────────────────┘
```

### 1.3 五大技术创新点

| 创新点 | 对应痛点 | 核心突破 |
|--------|----------|----------|
| **渐进式摘要记忆系统** | P3 | 自研 ConversationMemory，用 LLM 渐进压缩历史对话为摘要，替代简单截断，实现持久长期记忆 |
| **LLM 驱动的意图路由** | P2 | 让大模型先判定问题是否需要检索、目标手册是谁、查询关键词是什么，再执行精准检索 |
| **检索后 ReRank 二次精排** | P2 | 粗召回 + 精排序两阶段策略，Qwen3-Rerank 对 Top-K 结果重新打分排序，提升检索准确率 |
| **图文引用自动对齐** | P1/P4 | 检索结果中 <PIC:img_id> 标记与答案生成强制关联，回答中精确引用原文配图，杜绝张冠李戴 |
| **严格知识锚定机制** | P4 | 答案必须基于检索到的手册内容生成，Prompt 中明确"严禁编造或推测"，实现强幻觉抑制 |

## 三、核心技术实现

### 3.1 多模态信息解析（解决痛点 P1）

**问题**：用户上传的图片（产品故障照片、订单截图、错误代码界面）包含关键信息，纯文本模型无法理解。

**方案**：采用 **Qwen-VL-Max** 多模态大模型进行图文意图融合。

```
用户输入: 文本问题 + N 张 Base64 图片
                  │
                  ▼
┌──────────────────────────────────────────┐
│          ask_images() 视觉解析             │
│                                          │
│  构造视觉 Prompt:                          │
│  "用户上传了相关客服凭证图片，并提出了问题：  │
│   【{question}】。请结合图片，提取出与问题   │
│   相关的核心要素（如产品型号、指示灯状态、    │
│   错误代码、物流状态等），并用简短的文字概括  │
│   这些信息。"                             │
│                                          │
│  → 调用 dashscope.MultiModalConversation  │
│  → 返回图文融合后的增强文本                  │
└──────────────────────────────────────────┘
                  │
                  ▼
         增强后的文本问题（融合了图片信息）
         进入后续检索与生成管线
```

**设计考量**：
- 图片解析放在管线的**第一步**，确保后续检索和生成都能利用图片提取出的关键信息
- 视觉模型失败时降级返回空字符串，**不阻塞主流程**，保证系统鲁棒性
- 支持单次最多 3 张图片的批量解析

---

### 3.2 两阶段 RAG 精准检索（解决痛点 P2）

**问题**：40 本手册、20 万+ 字知识库，简单关键词匹配命中率低。

**方案**：设计 **"意图路由 → 粗召回 → 精排序"** 三级检索策略。

#### 3.2.1 阶段一：LLM 意图路由

在检索之前，先让 LLM 对用户问题进行智能分流：

```
用户问题 → DeepSeek-V4 路由分析 → JSON 输出:
{
  "target_doc": "健身追踪器手册",   // 目标手册名称
  "refined_query": "更换表带 尺寸 选项", // 去噪后的检索关键词
  "answer": "None"                   // 若可直接回答则输出答案
}
```

**路由规则**：
- **情况 A（直接回答）**：闲聊、问候、超出产品范围的问题 → 直接输出 answer，跳过检索
- **情况 B（触发检索）**：具体产品使用/故障问题 → 输出目标手册名 + 优化后的查询关键词

**关键设计**：
- `refined_query` 中去除了礼貌用语、语气助词，保留核心动词和名词
- `target_doc` 用于 Milvus 的**分区过滤**（`filter=f'doc_name == "{target_doc}"'`），大幅缩小检索范围，避免跨手册噪音

#### 3.2.2 阶段二：向量粗召回 + ReRank 精排

```
refined_query
      │
      ▼
text-embedding-v4 (1024维向量)
      │
      ▼
Milvus HNSW 检索 ─── filter: doc_name == target_doc
      │               metric: COSINE
      │               top_k: 5
      ▼
Top-5 候选 Chunks
      │
      ▼
Qwen3-Rerank 重排序 ─ instruct: "给定产品手册查询，检索出能够准确回答问题的段落。"
      │               top_n: 2
      ▼
Top-2 最相关段落 → 注入生成 Prompt
```

**检索性能优化**：
- **HNSW 索引**：M=16（连接数）, efConstruction=128（构建搜索范围），平衡精度与速度
- **COSINE 相似度**：适用于文本语义匹配场景
- **doc_name 过滤**：利用 Milvus 的标量过滤能力，在向量检索前先限定文档范围，准确率显著优于全库检索
- **ReRank 精排**：粗召回覆盖范围广（nprobe=10），精排序聚焦最相关（top_n=2），兼顾召回率和精确率

### 3.3 渐进式摘要记忆系统（解决痛点 P3）

**问题**：传统做法是截断早期对话（如只保留最近 K 轮），这会导致关键信息永久丢失。在多轮追问场景中，用户常说"那别的型号呢？""如果不舒适怎么办？"，这些指代需要完整的历史上下文才能理解。

**方案**：自研 **ConversationMemory** 类，实现基于 LLM 摘要的渐进式对话记忆。

#### 3.3.1 记忆结构

```
┌─────────────────────────────────────────────┐
│            ConversationMemory               │
│                                             │
│  ┌───────────────────────────────┐          │
│  │  summary: str                 │          │
│  │  "用户在咨询人体工学椅靠背    │          │
│  │   调节方法，已告知调节杆位置  │          │
│  │   和安全注意事项..."          │          │
│  │  (LLM 渐进式压缩的历史摘要)    │          │
│  └───────────────────────────────┘          │
│                                             │
│  ┌───────────────────────────────┐          │
│  │  messages: List[Dict]         │          │
│  │  [                           │          │
│  │    {"role":"user",            │          │
│  │     "content":"调节时有什么    │          │
│  │     安全注意事项？"},          │          │
│  │    {"role":"assistant", ...}, │          │
│  │    ... (最近 6 条)            │          │
│  │  ]                            │          │
│  └───────────────────────────────┘          │
│                                             │
│  max_recent = 6 (保留原始消息的窗口)         │
└─────────────────────────────────────────────┘
```

#### 3.3.2 摘要触发与合并策略

```
每次 add_turn(user_msg, assistant_msg):
  1. 追加新消息到 messages 列表
  2. 检查: len(messages) > max_recent 且 溢出部分 >= 2 条?
     ├── 否 → 不触发，等待下次
     └── 是 → 触发摘要压缩:
          ├── 将溢出部分（messages[:-max_recent]）取出
          ├── 构造合并 Prompt:【已有摘要】+【新增对话】
          ├── 调用 DeepSeek-V4 生成新摘要
          ├── 更新 self.summary
          └── 截断 messages 为最近 max_recent 条
```

**摘要 Prompt 设计要点**：
- 保留所有关键事实：产品名称、型号、问题类型、用户需求、已提供的方案
- 保留重要数字、步骤编号、错误代码、状态描述
- 丢弃寒暄、礼貌用语、重复内容
- 摘要控制在 300 字以内，确保注入 Prompt 时不超 Token 限制

#### 3.3.3 上下文注入

每次生成回答时，记忆上下文以如下格式注入 Prompt：

```
【对话历史摘要】:
用户在咨询人体工学椅靠背调节方法...

【近期对话】:
用户: 调节时有什么安全注意事项吗？
客服: 调节靠背时请注意...
```

**对比分析**：

| 策略 | 早期信息保留 | 上下文长度 | 指代理解 |
|------|-------------|-----------|---------|
| 简单截断（仅保留最近 K 轮） | ❌ 永久丢失 | 可控 | ❌ "它"无下文 |
| 全量保留（无压缩） | ✅ 完整 | 线性增长，终超 Token 限制 | ✅ 但有上限 |
| **渐进式摘要（本方案）** | ✅ 语义保留 | O(1) 可控 | ✅ 摘要覆盖关键实体 |

### 3.4 强幻觉抑制机制（解决痛点 P4）

**问题**：大模型在缺乏知识时倾向于"编造"答案，这在客服场景中是致命的——编造的操作步骤可能导致用户损坏产品或受伤。

**方案**：我们从 **Prompt 约束 + 知识锚定 + 降级兜底** 三个层面构建幻觉抑制体系。

#### 3.4.1 Prompt 层严格约束

```
qa_system_prompt 核心规则:
┌─────────────────────────────────────────────┐
│ 1. 幻觉抑制：你的回答必须完全基于参考内容，    │
│    严禁自己编造或推测。若未找到相关说明，      │
│    请坦诚告知。                              │
│                                             │
│ 2. 精准插图引用：参考内容中标注了图片位置      │
│    （格式如 <PIC:Manual02_11>），若原文该处    │
│    紧跟 <PIC:图片名>，你必须在回答的对应位置   │
│    原样带上！绝对不要漏掉或修改。              │
│                                             │
│ 3. 思维链拆解：若用户一次提问包含多个子问题，  │
│    逐步拆解，条理清晰一一对应作答。            │
└─────────────────────────────────────────────┘
```

#### 3.4.2 知识锚定机制

生成 Prompt 的结构确保了答案的"知识来源可追溯"：

```
【对话历史摘要】: ...（可选）
【参考手册内容】:                        ← 知识锚定源
  ---(段落1)---
  靠背调节杆位于座椅右侧下方...
  <PIC:Chair_03>                       ← 图片引用标记
  ---(段落2)---
  调节时请确保座椅无人乘坐...
  
【当前问题】:                           ← 用户问题
  人体工学椅的靠背怎么调节？
```

模型被强制约束在"参考手册内容"的范围内生成答案，**任何超出范围的信息都是违规的**。

#### 3.4.3 路由层的安全兜底

在意图路由阶段，系统识别出以下情况会**直接由 LLM 回答而不检索**：
- 闲聊问候（"你好""在吗"）
- 超出产品范围的问题（"这瓶饮料多少钱"）
- 知识库中不存在对应产品

这类问题的回答由路由 LLM 当场输出，**不经过检索-生成链路**，避免"强行检索 + 强行基于不相关内容生成"导致的幻觉。

### 3.5 双语知识库融合（解决痛点 P5）

**挑战**：知识库包含 20 本中文手册 + 20 本英文手册，用户可能用中文问英文产品的问题，或反之。

**方案**：

#### 3.5.1 统一向量空间建库

```
中文手册 (.txt) ──→ txt_split.py ──→ Chunks ──→ text-embedding-v4 ──→ Milvus
英文手册 (.txt) ──→ txt_split.py ──→ Chunks ──→ text-embedding-v4 ──→ Milvus
英文汇总手册 ────→ build_english_db.py (暴力拆分+自动识别doc边界) ──→ Milvus
```

所有手册（中英文）统一使用 `text-embedding-v4`（1024 维）向量化，存入同一个 Milvus Collection，**语义空间对齐**。

#### 3.5.2 路由 Prompt 中的语言一致规则

```
【核心语言规则 —— 最高优先级】
- 用户使用什么语言提问，你的所有输出必须使用相同的语言。
- 英文问题用英文关键词，中文问题用中文关键词。
- 注意：参考手册内容可能是中英混杂的，
  这不应影响你回答的语言选择——始终跟随用户的语言。
```

#### 3.5.3 英文汇总手册的智能拆分

英文知识库以单个汇总 JSON 文件提供，我们设计了 `build_english_db.py` 实现**自动边界识别**：
- 按图片文件名前缀（如 `Microwave_`, `Laptop_`）推断文档边界
- 自动拆分为 20 个独立英文手册
- 追加到已有中文 Milvus 库，实现**增量建库**

### 3.6 SSE 流式响应（体验优化）

**方案**：支持 `stream=true` 参数的 SSE（Server-Sent Events）流式响应路径。

```
generate_with_progress() 渐进式事件流:

event: step
data: {"step": "memory", "label": "正在加载历史对话..."}

event: step

event: step
```

## 四、数据与知识库构建

### 4.1 数据处理管线

```
原始手册数据 (txt + 插图文件夹)
        │
        ▼

        │

│  • HNSW 索引构建                    │

└───────────────────────────────────┘
```

### 4.2 Milvus Schema 设计

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INT64 (主键, 自增) | 唯一标识 |
| `doc_name` | VARCHAR(255) | 手册名称，用于标量过滤 |
| `content` | VARCHAR(65535) | 文本内容（含 `<PIC:img_id>` 标记） |
| `image` | ARRAY[VARCHAR(100)] | 关联图片 ID 列表 |

**索引配置**：
- 索引类型：HNSW（分层可导航小世界图）

## 六、模型选型与调用策略

| 模型 | 用途 | 选型理由 |
|------|------|----------|
| **DeepSeek-V4-Pro** | 意图路由、答案生成、摘要压缩 | 强推理能力，支持中文，性价比优于 GPT-4 |

| **text-embedding-v4** | 文本向量化 | 1024 维，中英双语语义对齐，适合混合语言知识库 |
| **Qwen3-Rerank** | 检索结果重排序 | 针对"给定查询，检索相关段落"的 instruct 调优，显著提升 Top-2 准确率 |
| **Qwen3.5-Flash** | 轻量任务（备用） | 低延迟，适合简单分类和快速响应场景 |

## 八、方案亮点总结

### 8.1 架构亮点

1. **四阶段流水线解耦**：感知、路由、检索、生成各自独立，易于单独优化和替换模型
2. **双路径输出**：同步 JSON + SSE 流式，适配不同客户端场景
3. **多模型协同**：不同模型负责不同子任务，各取所长（DeepSeek 推理 + Qwen 视觉 + DashScope Embedding/Rerank）

### 8.2 创新亮点

1. **渐进式摘要记忆**：自研 ConversationMemory，用 LLM 压缩替代简单截断，在 O(1) 空间内实现持久长期记忆
2. **LLM 意图路由先行**：在检索前由 LLM 判定"是否检索 + 目标手册 + 优化查询词"，大幅提升检索精准度
3. **图文引用自动对齐**：知识库中 <PIC:img_id> 标记 → Prompt 强制关联 → 回答中精确引用，打通图文全链路
4. **知识锚定式幻觉抑制**：分层设计（Prompt 约束 + 知识来源限定 + 路由安全兜底），而非仅依赖 Prompt 提示
5. **双语统一向量空间**：中英文手册统一 Embedding 模型，语义空间对齐，跨语言检索不受语言边界限制

### 8.3 工程亮点

1. **增量建库**：支持新手册追加而不重建全库，维护 `doc_names.json` 目录供路由 LLM 读取
2. **降级容错**：视觉解析失败 → 原问题继续；摘要生成失败 → 保留原始消息；LLM 调用异常 → 返回标准 500 JSON
3. **边界校验完备**：图片数量、格式、大小三重校验，401/400/500 错误码全覆盖
4. **代码注释覆盖**：关键算法（ConversationMemory、意图路由、检索管线）均附详细注释说明
